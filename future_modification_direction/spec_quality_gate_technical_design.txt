================================================================================
Spec Quality Gate 技术方案设计文档
智能需求质量关卡 — OpenSpec 改进方向 1
================================================================================

Context
-------

OpenSpec 当前工作流中，所有质量检查（verify skill）都发生在代码实现之后。
学术研究表明 30-50% 的软件返工可追溯到需求阶段的模糊、遗漏或矛盾。
Spec Quality Gate 在 artifact 创建阶段（编码前）植入 AI 驱动的质量检查，
尽早发现问题，大幅减少返工。

当前竞品（Amazon Kiro、GitHub Spec Kit、Tessl）均未提供需求质量评分能力，
这是 OpenSpec 的核心差异化机会。


================================================================================
0. 学术研究基础
================================================================================

本方案的每项质量检查均基于成熟的 RE/SE 学术研究。以下是核心理论支撑：

0.1 为什么需要 Quality Gate：返工成本经济学
------------------------------------------------

Boehm & Basili (2001) "Software Defect Reduction Top 10 List" [IEEE Software]
的经典研究表明：
- 40-50% 的软件项目总成本花在可避免的返工上
- 其中 70-85% 的返工成本可追溯到需求缺陷
- 需求阶段修复成本 1x → 集成测试阶段 21-78x → 生产环境 >1500x

核心论点：越早在 artifact 创建阶段发现需求问题，修复成本越低。
这是 Quality Gate 存在的经济学基础。

References:
- Boehm, B. & Basili, V. (2001). Software Defect Reduction Top 10 List.
  IEEE Software, 18(1), 135-137.
  https://www.cs.umd.edu/projects/SoftEng/ESEG/papers/82.78.pdf
- McConnell, S. (2001). An Ounce of Prevention. IEEE Software.
  https://stevemcconnell.com/articles/an-ounce-of-prevention/


0.2 检查什么：ISO/IEC/IEEE 29148:2018 质量框架
-------------------------------------------------

国际标准 ISO/IEC/IEEE 29148:2018 定义了需求的 9 个必要质量特征，
本方案的每项检查直接映射到这些特征：

  ISO 29148 质量特征     定义                           本方案对应检查
  ─────────────────────  ─────────────────────────────  ──────────────────────────────
  Unambiguous            需求只有一种解读方式           S-05, P-06, UC-07, D-07（模糊词扫描）
  Complete               定义了满足需要的全部能力/约束   S-01, S-07, UC-01, D-01, T-04
  Verifiable             需求的实现可被验证/测试         S-02 (WHEN/THEN), S-06, UC-08
  Correct                准确表述利益相关者需求         CC-01~CC-09（跨artifact一致性）
  Consistent             与其他需求不矛盾               CC-06 (Non-Goals冲突), CC-09
  Singular               每条需求只表述一个能力         T-03（单任务不超过2句）
  Feasible               技术上可实现                   D-02, D-03（决策有rationale）
  Necessary              确实需要的能力                 CC-02, CC-08（范围蔓延检测）
  Conforming             符合格式标准                   S-04（标题级别）, T-01（checkbox格式）

补充标准：IEEE 830-1998 定义 SRS 四个关键质量属性：
Completeness, Correctness, Preciseness, Consistency。
其中 Completeness 评估文档结构，其余三个评估功能需求。
这与本方案将检查分为"结构检查"（40%权重）和"内容检查"（40%权重）的设计一致。

References:
- ISO/IEC/IEEE 29148:2018. Systems and software engineering — Requirements engineering.
  https://cdn.standards.iteh.ai/samples/72089/62bb2ea1ef8b4f33a80d984f826267c1/ISO-IEC-IEEE-29148-2018.pdf
- IEEE 830-1998. Recommended Practice for Software Requirements Specifications.
  https://ieeexplore.ieee.org/document/720574
- Rajkumar, A. et al. (2024). Well-Formed Quality of System Requirements for
  Verifying to ISO 29148-2018: An NLP-Based Framework and Quantitative Metric.
  ASME IDETC-CIE 2024.
  https://www.researchgate.net/publication/385802396


0.3 如何检测模糊性：Requirements Smells
------------------------------------------

Femmer et al. (2017) "Rapid Quality Assurance with Requirements Smells"
[Journal of Systems and Software] 提出了 Requirements Smells 概念
（类比 Code Smells），定义了 8 类需求"异味"，并实现了自动检测原型 Smella：

- 检测方法：POS tagging（词性标注）+ Dictionary-based（字典匹配）
- 四类"异味词"：
  * Polysemy（多义词）
  * Subjective language（主观语言）
  * Ambiguous adverbs/adjectives（模糊副词/形容词）
  * Non-verifiable terms（不可验证术语）
- 在 3 个工业案例 + 1 个大学案例中验证

本方案的四级词汇分类（Hedge Words / Subjective Terms / Passive Uncertainty /
Scope Escape Words）直接基于 Femmer 的 Requirements Smells 分类体系。

后续扩展：
- Li et al. (2024) 将 Requirements Smells 用于可测试性度量，
  发现 smells 与需求可测试性强相关。
- Malardalen University (2025) 的工业实验显示 LLM 在需求歧义检测上
  达到实用水平，但需要领域适配。
- Multi-label classification (2025) 使用深度学习对需求 smells 进行多标签分类。

References:
- Femmer, H. et al. (2017). Rapid Quality Assurance with Requirements Smells.
  Journal of Systems and Software, 123, 98-113.
  https://www.sciencedirect.com/science/article/abs/pii/S0164121216000789
- Li, S. et al. (2024). Natural Language Requirements Testability Measurement
  Based on Requirement Smells. Neural Computing and Applications.
  https://link.springer.com/article/10.1007/s00521-024-09730-x
- Requirements Ambiguity Detection and Explanation with LLMs: An Industrial
  Study (2025). Malardalen University.
  https://www.ipr.mdu.se/pdf_publications/7221.pdf


0.4 Use Case 质量标准：Cockburn 方法论
-----------------------------------------

Cockburn (2001) "Writing Effective Use Cases" [Addison-Wesley] 定义了
Use Case 的结构化质量标准：
- 必须包含的字段：Primary Actor, Scope, Level, Preconditions, Trigger,
  Main Success Scenario, Extensions
- Main Scenario 步骤数：3-9 步（太少=太抽象，太多=应拆分为子用例）
- Extensions 穷举原则："brainstorm and exhaustively list the extension conditions"
- Actor 明确性：应为具名角色，非泛化"用户"

本方案的 UC-01 到 UC-08 直接基于 Cockburn 的质量准则。
OpenSpec 的 schema instruction 中已引用 Cockburn 方法论（"渐进式揭示"流程），
Quality Gate 将其从"建议"提升为"可度量的检查"。

Reference:
- Cockburn, A. (2001). Writing Effective Use Cases. Addison-Wesley.
  https://kurzy.kpi.fei.tuke.sk/zsi/resources/CockburnBookDraft.pdf


0.5 BDD 场景质量：WHEN/THEN 检查
------------------------------------

Binamungu et al. (2020) "Characterising the Quality of BDD Specifications"
[REFSQ 2020] 是首个系统研究 BDD 规格质量的学术工作：
- 可读性是从业者评价最高的 BDD 质量属性
- 目前尚无公认的 BDD 质量度量标准 — OpenSpec 的 Quality Gate 如定义
  一套 BDD 场景质量检查，本身即为学术贡献
- Given/When/Then 格式的核心价值在于将需求从模糊变为可执行的验收测试

本方案的 S-01（每个需求必须有 Scenario）、S-02（每个 Scenario 必须有 WHEN/THEN）
填补了 BDD 质量度量的空白。

References:
- Binamungu, L. et al. (2020). Characterising the Quality of BDD Specifications.
  REFSQ 2020.
  https://link.springer.com/chapter/10.1007/978-3-030-49392-9_6
- Olorunlomerue, A. (2024). Applying BDD to Improve Software Quality.
  World Journal of Advanced Research and Reviews.
  https://wjarr.com/sites/default/files/WJARR-2024-0332.pdf


0.6 跨 Artifact 一致性：Requirements Traceability
-----------------------------------------------------

跨 artifact 检查基于 Requirements Traceability 这一成熟研究领域：

- Zhao, Alhoshan, Ferrari et al. (2021) 的系统映射研究（404 篇论文）
  将 NLP4RE 任务分类，traceability link recovery 是最活跃方向之一
- 2024 年 systematic mapping study 显示 LLM 在需求-代码追溯上
  优于传统 IR 方法
- EU AI Act Article 11（2026.8 生效）要求高风险 AI 系统具备
  全链路可追溯性 — 为跨 artifact 一致性检查提供合规驱动力

本方案的 CC-01~CC-09 将学术界的 traceability link recovery 方法
应用于 OpenSpec 的多 artifact 工作流。

References:
- Zhao, L. et al. (2021). NLP for Requirements Engineering: A Systematic
  Mapping Study. ACM Computing Surveys, 54(3), 1-41.
  https://dl.acm.org/doi/10.1145/3444689
- Auxiliary Artifacts in Requirements Traceability: A Systematic Mapping
  Study (2025).
  https://arxiv.org/html/2504.19658v1
- Empirical Study on Requirement-to-Code Traceability Link Recovery (2024).
  https://www.sciencedirect.com/science/article/pii/S1319157824002076

================================================================================
1. 总体架构
================================================================================

设计原则
--------
- Warning by default, not blocking — 符合 OpenSpec "fluid not rigid" 哲学
- Claude IS the runtime — 所有检查通过 SKILL.md prompt 指令执行，无需额外代码运行时
- Mirror verify pattern — 输出格式、严重级别、报告结构与现有 verify skill 保持一致

新增文件
--------
  文件                                              作用
  ─────────────────────────────────────────────────  ──────────────────────────────
  .claude/skills/openspec-quality-gate/SKILL.md      核心 skill 定义（~200行 prompt 指令）
  .claude/commands/opsx/quality.md                   命令包装器
  openspec/config.yaml 扩展                          新增 quality: 配置段

调用方式
--------
- 按需调用：/opsx:quality 或 /opsx:quality <change-name>
- 自动检查（可选）：artifact 创建后自动运行，通过 config.yaml quality.auto_check: true 启用
- Apply 前关卡（可选）：quality.gate_before_apply: true 在实现前提示质量分数

2. 质量检查维度
================================================================================

2.1 逐 Artifact 检查
---------------------

Proposal 检查（9 项）

  ID    检查项                                               严重级别    权重
  ────  ───────────────────────────────────────────────────   ────────    ────
  P-01  Why 段落包含具体问题陈述（>=2 句）                    CRITICAL    15
  P-02  What Changes 每项指向具体行为，非抽象目标             WARNING     10
  P-03  至少定义了一个 New/Modified Capability                CRITICAL    20
  P-04  Capability 名称为 kebab-case 且有描述性               WARNING     5
  P-05  Impact 段落列出了受影响的代码路径/依赖                WARNING     10
  P-06  模糊词扫描（Why/What 段落）                          WARNING     15
  P-07  无开放性范围词（"etc.", "and more", "various"）       WARNING     10
  P-08  Breaking Changes 有明确标注                          INFO        5
  P-09  每个 Capability 在 Impact 中有对应提及                WARNING     10

Usecases 检查（8 项）

  ID     检查项                                                        严重级别    权重
  ─────  ─────────────────────────────────────────────────────────────  ────────    ────
  UC-01  每个 Use Case 包含所有必填字段                                 CRITICAL    20
  UC-02  Main Scenario 有 3-9 个步骤（Cockburn 推荐）                   WARNING     10
  UC-03  Primary Actor 是具名角色，非泛化"用户"                         WARNING     10
  UC-04  每个 Use Case 至少有一个 Extension（异常/替代流）               WARNING     15
  UC-05  Level 为 "User goal"（非全部 Summary/Subfunction）            INFO        5
  UC-06  Proposal 中每个 Capability 至少有一个对应 Use Case             CRITICAL    20
  UC-07  场景步骤中的模糊词扫描                                        WARNING     10
  UC-08  Success Guarantee 包含可观察/可度量的后置条件                   WARNING     10

Specs 检查（10 项）— 最高权重

  ID    检查项                                               严重级别    权重

  S-01  每个 Requirement 至少有一个 Scenario                  CRITICAL    20
  S-02  每个 Scenario 包含 WHEN 和 THEN 子句                  CRITICAL    15
  S-03  使用规范性语言（SHALL/MUST），非被动语态              WARNING     10
  S-04  Scenario 使用 #### 四级标题                           CRITICAL    5
  S-05  需求描述中的模糊词扫描                                WARNING     15
  S-06  缺少具体数值/阈值/可度量标准的需求                    WARNING     10
  S-07  仅有 Happy Path 场景，缺失错误/边界场景               WARNING     10
  S-08  MODIFIED Requirements 包含完整更新内容                CRITICAL    5

  D-02  每个 Decision 包含 Rationale 说明                     WARNING     15
  D-03  至少一个 Decision 包含评估过的替代方案                WARNING     10

  D-05  Decisions 引用了 spec requirement 或 capability 名称  WARNING     15

  D-07  决策和风险描述中的模糊词扫描                          WARNING     10

  ────  ───────────────────────────────────────────────────   ────────    ────

  T-06  任务描述中的模糊词扫描                                WARNING     15

3. 模糊词检测策略
================================================================================

3.1 四级词汇分类
-----------------

Level 1: Hedge Words — 几乎总是有问题
  might, maybe, perhaps, possibly, probably, sometimes, usually,
  generally, often, occasionally, typically, commonly, normally,
  as appropriate, as needed, if possible, where applicable,
  etc., and so on, and more, among others

Level 2: Subjective/不可度量词 — 需要量化
  fast, slow, quick, efficient, user-friendly, intuitive, simple,
  easy, adequate, sufficient, reasonable, appropriate, proper,
  good, bad, optimal, best, minimal, significant, substantial,
  several, many, few, some, various, numerous

Level 3: 弱化语气词 — 削弱需求强度
  should be able to, could potentially, is expected to,
  it would be nice, ideally, in theory, presumably,
  we think, we believe, we hope, we assume

Level 4: 范围逃逸词 — 暗示无界范围
  all relevant, everything, anything, comprehensive, complete coverage,
  handle all cases, support all, any scenario


3.2 上下文感知规则
-------------------

  规则                                  说明
  ────────────────────────────────────  ───────────────────────────────────────────
  Proposal Why 段落                     严重级别降为 INFO（叙述性文本允许一定模糊性）
  代码块 / 文件路径内                    跳过检测
  引号内文本                            跳过检测
  技术标识符（camelCase / kebab-case）   跳过检测
  模糊词后紧跟具体数值/范围              降为 INFO（如 "approximately 5ms"）
  config.yaml dictionary 中的词          跳过检测

4. 评分算法
================================================================================

4.1 单条 Requirement 评分
--------------------------

  RequirementScore = 100 - sum(penalty)
    CRITICAL issue = -25
    WARNING issue  = -10
    INFO issue     = -3
    Floor = 0

4.2 单个 Artifact 评分
------------------------

  ArtifactScore = StructuralScore * 0.4 + ContentScore * 0.4 + ConsistencyScore * 0.2

  - StructuralScore: 段落完整性、格式正确性
  - ContentScore: 模糊词、可测试性、边界覆盖
  - ConsistencyScore: 跨 artifact 对齐

4.3 整体 Change 评分
----------------------

  OverallScore = proposal * 0.15 + usecases * 0.15 + specs * 0.30
                 + design * 0.20 + tasks * 0.20

  Specs 权重最高（0.30），因为它是行为契约的核心。
  未创建的 artifact 不纳入计算，权重重新归一化。

4.4 阈值判定
--------------

  状态    条件                            含义
  ──────  ──────────────────────────────  ──────────────────────────
  PASS    Score >= 80 且无 CRITICAL       质量达标，可进入实现
  WARN    Score >= 60 且 CRITICAL <= 2    有改进空间，建议修复后实现
  FAIL    Score < 60 或 CRITICAL > 2      建议先修复关键问题

6. config.yaml 扩展
================================================================================

  # 新增 quality 配置段（所有字段可选，有默认值）
  quality:
    enabled: true              # 总开关（默认 true）
    auto_check: false          # artifact 创建后自动检查（默认 false）
    gate_before_apply: false   # apply 前提示质量分数（默认 false）
    block_on_fail: false       # 失败时强烈建议不继续（默认 false）

    thresholds:
      pass: 80                 # 及格线
      warn: 60                 # 警告线

    artifact_thresholds:       # 可选：按 artifact 覆盖阈值
      specs:
        pass: 85
        warn: 70

    checks:                    # 可选：按检查类别配置
      ambiguity:  { enabled: true, severity: WARNING }
      vagueness:  { enabled: true, severity: WARNING }
      completeness: { enabled: true, severity: CRITICAL }
      testability: { enabled: true, severity: WARNING }
      consistency: { enabled: true, severity: WARNING }

    dictionary: []             # 领域术语（不标记为模糊）
    custom_ambiguity_words: [] # 自定义模糊词
    ambiguity_exceptions: []   # 排除的模糊词

  openspec-ff-change/SKILL.md         Step 4 后新增: 若 auto_check: true，对所有
                                      artifact 运行质量检查，输出总分表
  openspec-propose/SKILL.md           Step 4 后新增: 同 ff
  openspec-apply-change/SKILL.md      Step 3 前新增: 若 gate_before_apply: true，
                                      检查 quality-report.md 或运行质量检查

每处修改约 5-10 行 prompt 指令，不影响现有逻辑。
